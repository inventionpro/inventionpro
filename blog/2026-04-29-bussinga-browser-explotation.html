2026-04-29 Bussinga browser explotation

One click, instant browser control exploit for Bussinga.

This is an issue first discovered last year but decided to make it a blog. First PoC in Apr 12, 2025

Discovery

As the creator of WXV, when I was first making it, I read through the source code of Napture and Bussinga (both messy in their own way, not like WXV is better) to know how everything was implemented.

WebX is extremly limited in all ways, while Bussinga implements extra features that allow extra posibilities to help devs make better pages, such as allowing normal CSS and extra Lua functions.

One of these "improvements" was how setting the content of elements is handled in Lua, normally .set_content() just sets text since Face (Creator of WebX) was lazy to implement an equivalent of innerHTML, but Bussinga allowed .set_content() to set HTML, not the limited version, full HTML.

So I tinkered a bit and made this PoC that can be found here, the relevant code:

get('l').set_content([[
  
]])

Vulnerability explanation

Very common XSS, except for the entire browser.

It sets the content of a element to be an image with an invalid src attribute, it fails to load and trggers the onerror.

Inside the onerror it accesses the window's parent (the browser), from which it can do anything (Sets the entire browser to a string in the PoC).

The code in the browser that allows this in src/lua.js:

set_content: (text)=>{
  c.innerHTML = text
},
set_contents: (text)=>{
  c.innerHTML = text
}

Current state

Currently still an issue in codingMASTER398 and WebX+ versions.

In the case of WXV, it isn't vulnerable by default, if Bussinga Lua is enabled it might be exploitable but the sandbox makes it harder and the web setting prevents any OS attacks.
